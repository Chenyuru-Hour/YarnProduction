# 软件需求规格说明书

## 1. 引言

### 1.1 编写目的

本文档旨在明确“取纱机生产数据监控系统”的功能、性能、数据及接口需求，为项目开发、测试及验收提供依据。\
预期读者包括项目管理人员、开发人员、测试人员及相关业务方。

### 1.2 范围

本系统是一个部署在本地电脑上的Web应用，用于实时采集16台取纱机的生产数据（机台编号、台位、纱卷重量），提供数据存储、查询、统计分析及实时监控功能。系统将长期保存所有数据，并支持按条件导出Excel报表。

### 1.3 定义与缩略语

| 术语 | 说明 |
| :--- | :--- |
| PLC | 可编程逻辑控制器，用于现场数据采集 |
| 取纱机 | 生产设备，每台设备包含多个台位 |
| 台位 | 取纱机上的具体工作工位 |
| 产能 | 单位时间内（通常按日）累计取纱总重量 |
| SQL Agent | SQL Server 代理，用于执行定时任务（如数据归档） |

### 1.4 参考资料

- 用户提供的项目需求描述
- 系统架构决策（ASP.NET Core + Blazor + SQL Server + Redis）
- SQL Server 分区表及代理作业技术文档

## 2. 项目概述

### 2.1 项目背景

车间现有16台取纱机，每台设备有多个台位，当前缺乏统一的数据采集与监控手段。需要建立一个本地化Web系统，自动记录每个纱卷的取纱数据，实时展示生产状态，并提供历史数据查询与导出能力，以支撑生产管理和绩效分析。

### 2.2 项目目标

- 自动采集PLC上传的取纱数据，存入数据库，保证数据完整性。
- 提供实时看板，动态显示各机台/台位的当前纱卷重量及当日累计产能。
- 支持按日期、机台编号、台位组合查询历史数据，并可导出为Excel文件。
- 实现数据的永久存储，并能应对每日数万条记录的增长，通过自动分区与归档机制维持数据库性能。

### 2.3 用户特点

- **操作员**：日常查看实时看板，查询和导出历史数据。
- **系统维护人员**：监控系统运行状态，处理异常（如PLC通信中断、磁盘空间告警）。

### 2.4 业务场景

- **数据采集**：PLC实时将取纱事件（取纱机编号、台位、重量、时间戳）发送给系统，系统记录至数据库。
- **实时监控**：车间大屏或操作员电脑上，看板以卡片形式展示所有取纱机的最新状态，重量和产能数据自动刷新。
- **历史查询**：用户选择日期范围、机台和台位，系统返回符合条件的数据列表，并支持导出。
- **数据维护**：系统后台自动进行分区扩展和历史数据归档，确保数据库长期稳定运行。

## 3. 功能性需求

### 3.1 PLC数据采集与存储

| 编号 | FR-01 |
| :--- | :--- |
| **描述** | 系统能够接收来自16台PLC上传的取纱数据，并将其存储到数据库中。 |
| **输入** | 取纱机编号（字符串）、台位（字符串）、纱卷重量（浮点数）、时间戳（日期时间）。 |
| **处理** | 对接收到的数据进行校验（如机台号范围、重量非负），若校验通过则写入数据库；若数据异常则记录错误日志，并触发告警（可选）。 |
| **输出** | 成功存储或失败记录。 |

### 3.2 历史数据查询

| 编号 | FR-02 |
| :--- | :--- |
| **描述** | 用户可根据日期范围、取纱机编号、台位筛选生产数据，以列表形式展示。 |
| **输入** | 开始日期、结束日期（必选）；取纱机编号（可选）；台位（可选）。 |
| **处理** | 系统根据筛选条件从数据库中检索数据，支持分页展示，默认按时间倒序排列。 |
| **输出** | 包含机台号、台位、重量、时间戳的数据列表。 |

### 3.3 数据导出为Excel

| 编号 | FR-03 |
| :--- | :--- |
| **描述** | 将查询结果导出为Excel文件，供用户存档或进一步分析。 |
| **输入** | 与FR-02相同的筛选条件。 |
| **处理** | 系统根据条件导出全部匹配数据（不分页），生成.xlsx格式文件。 |
| **输出** | Excel文件下载到用户本地。 |

### 3.4 实时看板

| 编号 | FR-04 |
| :--- | :--- |
| **描述** | 实时显示所有取纱机/台位的当前纱卷重量和当日累计产能。 |
| **输入** | 无用户主动输入，系统自动刷新。 |
| **处理** | 系统从缓存（Redis）中获取最新重量和当日累计数据，通过WebSocket（SignalR）推送到前端，前端自动更新展示。 |
| **输出** | 动态更新的看板界面，包含每个机台、每个台位的两项指标。 |

### 3.5 自动分区与归档

| 编号 | FR-05 |
| :--- | :--- |
| **描述** | 系统需自动管理数据表的存储空间，按月分区，并将超过指定时间（如12个月）的数据迁移至归档表，确保主表大小可控。 |
| **处理** | 由SQL Server Agent定时作业执行：每月创建新分区边界；每日/每月将超过12个月的数据从主表迁移至归档表。 |
| **输出** | 分区表自动扩展；历史数据成功归档；操作日志记录。 |

## 4. 非功能性需求

### 4.1 性能需求

- **数据采集吞吐量**：系统应能处理每日数万条记录（峰值约5条/秒），数据入库延迟不超过1秒。
- **查询响应时间**：在百万级数据量下，带条件查询（日期范围+机台）的响应时间应小于3秒；导出操作应能在合理时间内完成（如10万条数据导出时间小于30秒）。
- **实时看板延迟**：数据从PLC采集到前端展示的端到端延迟不超过2秒。
- **并发访问**：支持至少10个并发用户同时进行查询操作，不影响数据采集。

### 4.2 可靠性

- 系统应作为Windows服务运行，开机自启，具备异常自动重启能力。
- 数据采集服务应具备PLC通信故障重试机制，通信恢复后自动恢复采集。
- 数据库应定期备份（由DBA或SQL Agent作业执行），防止数据丢失。

### 4.3 安全性

- 系统应部署在本地网络，默认仅允许局域网内访问；可配置Windows身份验证或简单用户认证。
- 数据库连接字符串默认sa账户或弱密码。

### 4.4 可维护性

- 系统日志应记录关键操作（数据采集异常、归档任务执行、用户登录等），采用滚动文件方式存储，便于排查问题。
- 提供配置界面（或配置文件）用于修改PLC连接参数、采集频率等，无需重新编译代码。
- 数据库分区和归档作业应提供执行状态监控，失败时能发送告警（如邮件）。

### 4.5 数据容量

- 主表应通过分区方式将数据分散存储，单分区数据量控制在百万级以内。
- 归档表可独立存放历史数据，不影响日常查询性能。
- 数据库文件总容量需根据硬件规划，要求支持至少5年的数据存储（按每日5万条估算，约9GB/年）。

## 5. 数据需求

### 5.1 数据实体

- **生产记录表（ProductionRecords）**

| 字段名 | 类型 | 描述 |
| :--- | :--- | :--- |
| Id | bigint | 自增主键 |
| MachineId | nvarchar(50) | 取纱机编号（1~16） |
| StationId | nvarchar(50) | 台位标识 |
| BobbinWeight | decimal(18,3) | 纱卷重量(kg) |
| Timestamp | datetime2(0) | 采集时间 |

- **归档表（ProductionRecords_Archive）**\
结构与主表相同，用于存储超过12个月的数据。

- **归档日志表（ArchiveLog）**

| 字段名 | 类型 | 描述 |
| :--- | :--- | :--- |
| Id | int | 自增主键 |
| ArchiveDate | datetime | 归档执行时间 |
| RecordsArchived | int | 归档记录数 |
| Status | nvarchar(20) | 执行状态（成功/失败） |
| ErrorMessage | nvarchar(max) | 错误信息（可选） |

### 5.2 数据存储策略

- 主表使用 **分区表**，按 Timestamp 字段按月分区，分区函数边界动态扩展。
- 归档作业定期将主表中 Timestamp < DATEADD(month, -12, GETDATE()) 的数据迁移至归档表，并删除原数据（使用 DELETE 或分区切换）。
- 归档表不设分区，但应建立必要索引以支持偶尔的历史查询。

### 5.3 数据保留

- 数据永久存储，不主动删除。
- 用户可通过查询功能访问所有历史数据（主表+归档表），系统需在查询时合并两个表的数据（或提供切换选项）。

## 6. 外部接口需求

### 6.1 用户界面

- Web界面应兼容主流浏览器（Chrome, Edge, Firefox）。
- 实时看板采用卡片式布局，清晰显示16台机器的状态。
- 查询页面提供日期选择器、下拉框（机台、台位）、查询按钮、导出按钮。
- 支持响应式设计，适应不同分辨率。

### 6.2 硬件接口

- **PLC通信**：系统需通过以太网与PLC设备通信，具体协议（如Modbus TCP、S7、OPC UA）取决于现场PLC品牌，应抽象为可配置的驱动接口。
- **数据采集频率**：默认为1秒/次，可配置。

### 6.3 软件接口

- 数据库：使用 SQL Server（2019及以上），通过 Entity Framework Core 访问。
- 缓存：使用 Redis（版本5.0+），用于存储最新重量和当日累计产能。
- 导出：使用 EPPlus 库生成 Excel 文件。

## 7. 约束与假设

### 7.1 技术约束

- 开发语言：C#（.NET 8）
- 前端框架：Blazor Server
- 数据库：SQL Server 2019 Developer / Express / Standard
- 缓存：Redis
- 部署环境：Windows 10/11 或 Windows Server 2019+，需安装 .NET 8 Runtime 和 SQL Server。

### 7.2 假设

- PLC 设备能够稳定提供数据，通信协议为已知（或可模拟）。
- 网络环境稳定，服务器与PLC之间网络可达。
- 用户具备基本的计算机操作能力。
- 硬件资源（CPU、内存、磁盘）满足系统运行需求（建议至少4核CPU、8GB内存、100GB可用磁盘空间）。
