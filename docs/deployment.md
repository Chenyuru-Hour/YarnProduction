# 架构设计说明书

## 1. 引言

### 1.1 文档目的

本文档旨在描述“取纱机生产数据监控系统”的软件架构，包括技术选型、模块划分、数据流、部署模型及关键设计决策，为开发、测试、部署及后续维护提供统一的架构视图。

### 1.2 范围

本文档覆盖系统的主要架构元素，包括：
- 系统整体结构
- 技术栈及选型理由
- 模块划分与职责
- 数据架构（数据库设计、分区与归档）
- 部署架构（本地Windows服务）
- 关键非功能性需求实现方案

### 1.3 定义与缩略语

| 术语 | 说明 |
| :--- | :--- |
| PLC | 可编程逻辑控制器，现场数据源 |
| Blazor Server | .NET 框架，用于构建交互式Web UI，在服务器端运行组件，通过SignalR实时通信 |
| SignalR | ASP.NET Core 实时通信库，用于双向推送数据 |
| Redis | 内存数据库，用作缓存实时数据 |
| EF Core | Entity Framework Core，ORM框架 |
| SQL Server Agent | SQL Server 组件，用于定时执行作业（如数据归档） |

## 2. 架构概述

### 2.1 设计目标

- **数据持久化**：永久存储所有采集数据，每日数万条记录。
- **实时性**：PLC数据从采集到前端展示延迟 < 4秒。
- **可维护性**：自动化分区与归档，保证长期运行性能。
- **易部署**：单一可执行文件，可注册为Windows服务，开机自启。
- **可扩展性**：支持增加新PLC设备，支持未来数据量增长。

### 2.2 整体架构

系统采用 **分层架构 + 微服务化后台任务** 的设计，划分为以下层次：

1.  **采集层**：通过PLC驱动抽象，与16台PLC通信获取实时数据。
2.  **数据层**：SQL Server 负责持久化存储，Redis 缓存最新重量及当日累计产能(总卷数+总重量)。
3.  **应用层**：ASP.NET Core 应用，包含：
    - 后台采集服务（BackgroundService）
    - SignalR Hub，用于实时推送
    - Web API 控制器，提供查询、导出接口
4.  **表示层**：Blazor Server 前端，实现实时看板和查询导出界面。
5.  **基础设施**：Windows 服务宿主，SQL Agent 定时任务，日志记录。

## 3. 技术架构

### 3.1 技术栈选型

| 组件 | 技术选择 | 理由 |
| :--- | :--- | :--- |
| 后端框架 | ASP.NET Core 8 | 跨平台、高性能、内置依赖注入，支持Windows服务部署 |
| 前端框架 | Blazor Server | 全栈C#，降低前后端割裂；内置SignalR，实时看板实现简单 |
| 实时通信 | SignalR | 与Blazor Server无缝集成，WebSocket长连接，保证低延迟推送 |
| ORM | Entity Framework Core 8 | 简化数据访问，与SQL Server配合良好，支持LINQ查询 |
| 关系数据库 | SQL Server 2019 | 企业级功能，支持分区表，Agent作业自动归档 |
| 缓存 | Redis 5+ | 高性能内存数据库，存储实时指标，减少数据库查询压力 |
| Excel导出 | EPPlus | 无需Office组件，性能优秀，支持大数据量 |
| 日志 | Serilog | 结构化日志，可输出至文件、控制台，便于运维 |
| PLC通信 | 抽象接口 + 具体驱动（如S7.Net, NModbus） | 隔离硬件差异，支持模拟测试，便于切换品牌 |
| 部署 | Windows 服务 | 利用.NET的WindowsService支持，实现无登录自启动 |

### 3.2 模块划分

#### 3.2.1 采集模块
- **职责**：周期性读取PLC数据，校验并写入数据库和Redis，触发SignalR推送。
- **实现**：`BackgroundService` 子类 `PlcCollectorService`，通过依赖注入获取各组件。
- **关键点**：异常重试、并行采集16台设备、批量写入数据库（使用 EF Core.BulkExtensions）。

#### 3.2.2 实时推送模块
- **职责**：管理WebSocket连接，将采集到的数据实时推送给所有连接的客户端。
- **实现**：SignalR Hub `ProductionHub`，由采集服务调用 `IHubContext` 广播数据。
- **关键点**：区分不同客户端（如需推送特定机台），可使用分组。

#### 3.2.3 查询与导出模块
- **职责**：提供RESTful API，支持历史数据分页查询和Excel导出。
- **实现**：`ProductionController`，使用EF Core执行查询，EPPlus生成Excel。
- **关键点**：对于跨分区/归档表的数据查询，需合并两个表的结果（可在应用层或使用视图）。

#### 3.2.4 数据持久化模块
- **职责**：定义数据库上下文、实体映射，处理分区表和归档逻辑。
- **实现**：`AppDbContext`，通过 `OnModelCreating` 配置表映射；归档由SQL Agent作业执行存储过程完成。

#### 3.2.5 缓存模块
- **职责**：读写Redis，存储最新重量和当日累计产能。
- **实现**：`IDistributedCache` 封装，提供扩展方法（如 `StringIncrementAsync`）。
- **关键点**：每日累计使用原子自增，避免并发问题；设置合理过期时间（如7天）自动清理。

#### 3.2.6 后台任务模块
- **职责**：除采集服务外，还包括分区扩展、归档等辅助后台任务（可选，可由SQL Agent承担）。
- **实现**：若在应用内实现，则创建独立的 `BackgroundService`；推荐利用SQL Agent执行T-SQL，与应用解耦。

### 3.3 数据流

1.  **数据采集流**：PLC → 采集服务 → 数据库（批量插入） → Redis（更新最新值+累加当日产能） → SignalR（推送实时数据）
2.  **实时看板流**：浏览器 → SignalR 连接 → 接收推送 → 更新UI → 初始加载时调用API从Redis获取当前值
3.  **查询导出流**：浏览器 → Web API → 数据库（主表+归档表） → 返回JSON或生成Excel

## 4. 数据架构

### 4.1 数据库设计

**主表 `ProductionRecords`**
- 采用 **分区表**，按 `Timestamp` 字段按月分区，提高查询和维护效率。
- 分区函数和方案动态扩展（由SQL Agent作业每月添加新边界）。

**归档表 `ProductionRecords_Archive`**
- 结构与主表相同，但不分区，用于存储超过12个月的历史数据。
- 定期由SQL Agent作业将主表中过期数据迁移至此表，并从主表删除。

**日志表 `ArchiveLog`**
- 记录每次归档的执行时间、记录数、状态等，便于监控。

### 4.2 缓存设计

- **最新重量**：`latest:{MachineId}:{StationId}` → 重量值（字符串）
- **当日累计**：`daily:{yyyy-MM-dd}:{MachineId}:{StationId}` → 累计重量（浮点字符串）
- **过期策略**：每日累计键保留7天，最新重量键不设过期，由应用主动更新覆盖。

### 4.3 数据归档策略

- **时间阈值**：`Timestamp < DATEADD(month, -12, GETDATE())`
- **执行频率**：每天凌晨2点（通过SQL Agent作业）
- **迁移方法**：使用 `INSERT INTO ... SELECT` + `DELETE` 或利用分区切换（若按分区边界归档）。为确保性能，建议使用分区切换（需将分区对应到单独文件组，然后 `SWITCH` 到归档表）。
- **注意事项**：归档期间应避免对主表的写操作（或使用低优先级锁定）。

## 5. 部署架构

### 5.1 部署环境

- **操作系统**：Windows 10/11 或 Windows Server 2019+（建议64位）
- **运行时**：.NET 8 Runtime（如发布为自包含则无需安装）
- **数据库**：SQL Server 2019 及以上（Developer/Express/Standard）
- **Redis**：Windows 版 Redis（或WSL2/Linux），作为服务运行

### 5.2 部署形态

- **Web应用**：发布为单文件可执行程序，通过 `UseWindowsService()` 注册为Windows服务。
- **数据库**：安装在同一台机器或独立服务器，通过连接字符串指定。
- **PLC通信**：通过以太网与PLC设备连接，需确保网络畅通。

### 5.3 部署流程

1.  安装 SQL Server 并还原/创建数据库，执行分区创建脚本。
2.  安装 Redis 并配置为服务。
3.  发布应用，将文件复制到目标目录。
4.  配置 `appsettings.json`（数据库连接字符串、Redis地址、PLC驱动参数）。
5.  使用 `sc create` 注册服务，设置启动类型为自动。
6.  启动服务，验证应用可访问（默认端口 5000）。

### 5.4 安全配置

- 应用绑定 `localhost` 或内网IP，通过Windows防火墙限制访问。
- 启用 Windows 身份验证（可选），或使用简单登录页面。
- 数据库使用混合认证，创建专有账户，限制仅必要权限。

## 6. 关键设计决策

**6.1 为何选择 Blazor Server 而非前端框架分离？**
- **优点**：全栈C#，共享代码（实体、验证），开发效率高；内置SignalR支持，无需编写JavaScript。
- **权衡**：服务器占用较多内存（每个用户一个电路），但用户量小（≤10并发），可接受。

**6.2 为何使用 Redis 缓存实时数据？**
- 实时看板需要频繁读取最新重量和当日累计，若直接查询数据库，会带来较大负载。
- Redis 提供原子操作和高速读写，适合高频更新场景。

**6.3 为何使用 SQL Server 分区表 + Agent 归档？**
- 数据永久存储，量级大，必须控制主表大小。
- SQL Server 分区表可高效管理数据窗口（如按月分区），归档作业解耦应用逻辑，利用数据库原生能力更稳定。

**6.4 后台采集服务为何使用 BackgroundService？**
- 与ASP.NET Core 应用集成在同一进程，部署简单；利用依赖注入，易于测试和切换PLC驱动。

**6.5 为何不采用消息队列缓冲采集数据？**
- 当前数据量（每秒约5条）不大，直接写入数据库可满足性能；若未来扩展至更高频率，可引入RabbitMQ等中间件。

## 7. 质量属性

### 7.1 性能
- 采集写入：批量插入（每1秒累积一次数据），减少数据库IO。
- 查询：利用索引，分区裁剪，查询优化器自动过滤分区。
- 导出：流式读取，避免一次性加载全部数据到内存。

### 7.2 可靠性
- 采集服务异常捕获，记录日志，不中断循环。
- PLC通信重试策略（Polly库）。
- 数据库连接池管理，自动恢复。
- Windows服务失败自动重启（服务恢复设置）。

### 7.3 可扩展性
- 增加新PLC：仅需配置新设备地址，无需修改代码（PLC驱动可配置多端点）。
- 增加新台位：数据模型已支持任意台位标识，无需变更。
- 数据库扩容：可升级至标准版，增加文件组，或迁移至独立服务器。

### 7.4 可维护性
- 日志分级输出，支持按日期滚动。
- 配置集中管理，支持环境变量覆盖。
- 提供健康检查接口（`/health`），供监控系统调用。

## 8. 风险与应对

| 风险 | 应对措施 |
| :--- | :--- |
| PLC通信不稳定，数据丢失 | 启用重试机制，记录失败点；增加本地缓存暂存未写入数据；定期校验数据完整性。 |
| 数据库写入瓶颈 | 使用批量写入，考虑异步队列；对分区表优化，避免阻塞。 |
| 磁盘空间不足 | 监控磁盘使用率，设置告警；定期检查归档表大小，必要时压缩或清理（用户明确不删除，但可压缩归档表）。 |
| Redis 内存溢出 | 设置最大内存限制，配置LRU淘汰策略；合理设置键过期时间。 |
| SQL Agent 作业失败 | 作业中记录错误日志，配置邮件通知；定期检查作业历史。 |
| 实时看板延迟过高 | 优化SignalR消息大小，仅推送变更数据；前端做节流，避免频繁渲染。 |
